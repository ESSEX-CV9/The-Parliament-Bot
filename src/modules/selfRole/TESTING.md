# SelfRole 综合管理系统 - 端到端测试流程（仅测试环境）

本文档提供一套**可重复、可快速验证**的测试流程，用于覆盖 SelfRole 模块的完整闭环能力：

- 申请生命周期：预留名额 / 重复申请拦截 / 撤回 / 过期释放
- 审核投票闭环：通过 / 拒绝 / 冷却期（如配置）
- 配套身份组（bundle roles）发放
- grant 落库与生命周期调度：周期询问（留任/退出）/ onlyWhenFull / 强制清退
- DM 失败告警与管理员介入指引
- 一致性巡检：面板丢失 / 结束 grant 角色残留

> 注意：本文涉及的 `/自助身份组申请-测试` 为“测试/运维辅助命令”。
> - 该命令必须通过环境变量显式开启。
> - **正式环境不要启用**。

---

## 0. 测试环境快速启动

### 0.1 环境变量（建议用 `.env.test.sample`）

建议直接复制 `.env.test.sample` 为 `.env`，然后填写真实值：

- `DISCORD_TOKEN`：机器人 token
- `CLIENT_ID`：应用/机器人 Application ID
- `GUILD_IDS`：命令注册目标服务器列表（逗号分隔；测试建议只填测试服）

测试环境建议开启：

- `SELF_ROLE_ENABLE_TEST_COMMANDS=true`
  - 启用 `/自助身份组申请-测试`
- `STRICT_COMMAND_SYNC=true`
  - 启动时强制同步命令，任意目标服务器同步失败则直接中止启动
- `CLEAR_GLOBAL_COMMANDS=true`（仅在需要时开启一次即可）
  - 清空该应用的 **全局命令（Global Commands）**，避免同一个 bot/应用曾用于其它项目时遗留的全局命令干扰测试
  - 注意：全局命令的删除/传播可能存在 Discord 侧缓存延迟（通常可达约 1 小时）

### 0.2 安装与启动

```bash
npm install
npm start
```

启动日志中应看到类似信息：

- 已初始化 SQLite
- `SELF_ROLE_ENABLE_TEST_COMMANDS=true`（若开启）
- `Start refreshing ... commands for configured guilds...`

如果 `STRICT_COMMAND_SYNC=true` 且命令同步失败，启动会直接退出；请根据日志检查 `CLIENT_ID` / `GUILD_IDS` / 网络与权限。

### 0.3 邀请机器人与权限/层级

1) 邀请链接需要包含 scopes：
- `bot`
- `applications.commands`

2) 机器人需要具备（至少）以下权限：
- Manage Roles（管理身份组）
- Send Messages（发消息）
- Embed Links（发送 Embed）
- Read Message History（读取历史消息）
- Manage Messages（用于禁用旧面板组件/更新消息，推荐）

3) **身份组层级**：机器人自身的最高身份组必须高于：
- 需要授予/移除的岗位身份组（主身份组）
- 需要一并授予/移除的配套身份组（bundle roles）

否则会出现“授予失败/移除失败”，并导致巡检或告警。

## 1. 测试服资源准备（一次性）

建议准备至少 3 个测试账号（或找人协助）：

- 申请人 A：用于申请
- 申请人 B：用于容量/满员/预留名额测试
- 审核员 R：拥有“审核员”身份组，负责投票
- 管理员 Admin：用于配置、创建面板、查看/解决告警

建议创建如下身份组（示例命名，可自行替换）：

- `岗位A`（用于“需要审核 + 容量 + 生命周期”完整链路测试）
- `岗位A-配套1`（用于 bundle roles 测试）
- `审核员`（用于投票权限）
- （可选）`前置身份组`（用于 prerequisite 测试）
- （可选）`岗位B`（用于“直授”路径测试：不配置审核）

建议创建如下频道：

- `#岗位报名`：放用户申请面板
- `#岗位审核`：投票面板生成频道
- `#岗位报告`：生命周期报告/告警提示
- （可选）`#活跃度统计`：用于活跃度条件测试

---

## 2. 配置与面板创建

### 2.1 配置岗位A（推荐：带容量 + 审核 + 配套身份组 + 生命周期）

1) 基础配置（容量/活跃度等）：

- `/自助身份组申请-配置身份组 基础配置`
  - 目标身份组：`岗位A`
  - 人数上限：建议 `1` 或 `2`（便于测试满员/onlyWhenFull）
  - 统计频道/活跃度阈值：可先不配（后续可选测试再添加）

2) 审核配置：

- `/自助身份组申请-配置身份组 审核配置`
  - 审核频道：`#岗位审核`
  - 支持票阈值：`1`
  - 反对票阈值：`1`
  - 审核员身份组1：`审核员`
  - （可选）被拒后冷却天数：例如 `1`

3) 配套身份组配置（bundle roles）：

- `/自助身份组申请-配置身份组 配套身份组配置`
  - 目标身份组：`岗位A`
  - 配套身份组1：`岗位A-配套1`

4) 周期清退配置（grant 生命周期）：

- `/自助身份组申请-配置身份组 周期清退配置`
  - 目标身份组：`岗位A`
  - 启用：true
  - 询问周期天数：建议先填 `0`（测试用“立即触发询问”强制触发，不必等待）
  - 强制清退天数：建议先填 `0`（测试用“立即触发强制清退”强制触发）
  - 满员才执行：先设 false；后续再单测 onlyWhenFull
  - 报告频道：`#岗位报告`

5)（可选）申请理由配置（用于测试 modal 与长度校验）：

- `/自助身份组申请-配置身份组 申请理由配置`
  - 目标身份组：`岗位A`
  - 模式：`必需` 或 `可选`
  - 最小长度 / 最大长度：按需填写

### 2.2（可选）配置岗位B（直授路径测试，不配置审核）

如果你希望覆盖“无需审核、直接授予”的逻辑：

1) 创建一个 `岗位B`
2) 只做：
- `/自助身份组申请-配置身份组 基础配置`（可选设置人数上限/描述等）
- （可选）`配套身份组配置`
- （可选）`申请理由配置`

不要对 `岗位B` 执行“审核配置”，这样它会走**直授**路径。

### 2.3 创建用户面板与管理面板

- 在 `#岗位报名`：执行 `/自助身份组申请-创建自助身份组面板`
- 在管理员频道：执行 `/自助身份组申请-创建管理面板`

验证点：

- 用户面板包含岗位列表与 `【岗位状态】` 区块
- 新面板按钮 customId 为 `sr2_apply_button`
- 如果数据库记录中有旧面板，会被停用（组件清空），避免重复入口

## 3. 面板刷新与展示验证

手动触发一次刷新，确认统计区块可更新：

- `/自助身份组申请-测试 刷新面板`

验证点：

- `【岗位状态】` 中的“现任/空缺/待审核”能随申请/通过/撤回/过期变化

## 4. 申请生命周期（预留名额 / 重复拦截 / 撤回 / 过期）

以下建议以 `岗位A`（需要审核 + 容量）为主测对象。

### 4.1 创建申请 + 预留名额占用

用申请人 A：

1) 点击用户面板按钮，选择 `岗位A`
2) 若通过前置校验，将创建投票面板并占用预留名额

验证点：

- `#岗位审核` 出现投票面板
- 用户面板 `待审核` +1，`空缺` -1
- 投票面板 embed 中包含：配套身份组信息（如已配置）、容量统计（现任/上限/空缺/待审核）

### 4.2 重复申请拦截（同用户同岗位）

申请人 A 再次尝试申请 `岗位A`。

验证点：

- 被提示“正在审核中”，并提示可用 `/自助身份组申请-撤回申请`
- 不会重复创建新的投票面板

### 4.3 容量满员拦截（含 pending 预留名额）

前置：将 `岗位A` 人数上限设为 1（或 2），然后制造“占满”的状态：

- 情况 1：让 1 人已持有岗位A（通过审核获得）
- 情况 2：现任 0，但 pending 预留名额为 1

再让申请人 B 申请 `岗位A`。

验证点：

- 被提示“已满员（现任 X/上限，待审核 Y）”，无法申请

### 4.4 撤回 pending 申请

用申请人 A 执行：

- `/自助身份组申请-撤回申请 目标身份组:岗位A`

验证点：

- 投票面板按钮被禁用/标记为已撤回
- 面板统计：`待审核` -1，`空缺` +1（释放预留名额）

### 4.5 过期释放（测试命令）

先重新创建一条 pending（同 4.1），然后执行：

- `/自助身份组申请-测试 强制过期申请 目标用户:<申请人A> 目标身份组:岗位A`

验证点：

- 投票面板被标记为“已过期”，按钮禁用
- 面板统计恢复：`待审核` -1，`空缺` +1

也可以执行一次扫描（不修改数据，只跑扫描逻辑）：

- `/自助身份组申请-测试 检查申请过期`

---

## 5. 审核投票闭环（通过/拒绝/冷却/DM失败告警）

### 5.1 通过（授予主身份组 + 配套身份组 + grant 落库）

1) 申请人 A 发起 `岗位A` pending
2) 审核员 R 在投票面板点“支持”直至达到阈值

验证点：

- 申请人 A 获得：`岗位A` + `岗位A-配套1`
- 申请记录被终结，预留名额释放
- grant 落库（测试命令查看）：
  - `/自助身份组申请-测试 查看grant 目标用户:<申请人A> 目标身份组:岗位A`

### 5.2 拒绝（含冷却期可选）

1) 申请人 A 再次发起 pending
2) 审核员 R 点“反对”直至达到阈值

验证点：

- 申请记录被终结，预留名额释放
- 如果配置了“被拒后冷却天数”，再次申请会提示仍在冷却期

### 5.3 审核结果 DM 失败 -> 异常报告（可选，但强烈建议测）

让申请人 A 关闭私信（或禁止来自服务器成员的私信），然后重复 5.1/5.2。

验证点：

- 投票面板所在频道会出现**显眼的错误报告消息（Embed）**
  - 带按钮：**✅ 标记为已处理**
- 告警类型：`application_result_dm_failed`
- 点击“✅ 标记为已处理”后：按钮被禁用，告警记录被标记 resolved（用于去重/追踪）

## 6. 直授路径（无需审核）验证（可选）

对 `岗位B`（未配置审核）的申请将走直授：

1) 申请人 A 点击面板申请 `岗位B`

验证点：

- 立即获得 `岗位B`（以及 bundle roles 如配置）
- grant 落库：`/自助身份组申请-测试 查看grant ...`
- 面板统计刷新（可手动 `/自助身份组申请-测试 刷新面板` 验证）

## 7. grant 生命周期（询问/退出/强制清退/onlyWhenFull/DM失败告警）

前置：必须存在 active grant（建议先完成 5.1 或 6）。

### 7.1 触发询问（测试命令）

- `/自助身份组申请-测试 立即触发询问 目标用户:<申请人A> 目标身份组:岗位A`

说明：该命令会把该 grant 的 `nextInquiryAt` 设置到过去并立刻执行一次生命周期 tick。

验证点：

- 申请人收到 DM，包含“留任/退出”按钮
- 报告频道（如配置）出现询问相关报告

### 7.2 点击“留任”

在 DM 点击“留任”。

验证点：

- renewal session 状态变为 responded
- grant.last_decision 更新为 stay
- 下一次询问时间被推进（如 inquiryDays>0）

### 7.3 点击“退出”

再次触发询问（7.1），在 DM 点击“退出”。

验证点：

- 成员移除【主身份组 + 配套身份组】
- grant 结束（ended_reason=user_exit）
- 报告频道出现退出报告

### 7.4 触发强制清退（测试命令）

如果 7.3 已退出，需要先重新获得 active grant。

- `/自助身份组申请-测试 立即触发强制清退 目标用户:<申请人A> 目标身份组:岗位A`

说明：该命令会把 `forceRemoveAt` 设置到过去并立刻执行 tick。

验证点：

- 成员移除【主身份组 + 配套身份组】
- grant 结束（ended_reason=force_remove）
- 报告频道出现强制清退报告

### 7.5 onlyWhenFull 行为验证（强烈建议测）

目标：验证“未满员不执行动作，但计时继续；满员后可能立即补执行，并显示延后提示”。

步骤建议：

1) 将 `岗位A` 的 `周期清退配置` 中 `满员才执行` 设为 true
2) 将人数上限设为 1
3) 确保当前**不满员**（现任+pending 预留 < 1）
4) 执行 `/自助身份组申请-测试 立即触发询问 ...`

验证点：

- 此时不会发送 DM（因为 onlyWhenFull 且未满员）

5) 让岗位达到“满员”状态（例如让 1 人持有岗位A，或制造 pending 预留）
6) 执行 `/自助身份组申请-测试 执行生命周期`

验证点：

- 会补执行询问/清退动作
- 文案中会体现“延后执行”语义（提示是因 onlyWhenFull 延迟到满员后才执行）

### 7.6 生命周期 DM 失败 -> 异常报告

让目标用户关闭私信，然后执行：

- `/自助身份组申请-测试 立即触发询问 ...`

验证点：

- 报告频道出现**显眼的错误报告消息（Embed）** + 操作建议
  - 带按钮：**✅ 标记为已处理**
- 告警类型：`lifecycle_dm_inquiry_failed`
- grant 被标记 `manual_attention_required=1`

类似地，强制清退通知 DM 失败会形成：

- `lifecycle_dm_force_remove_failed`

## 8. 异常报告处理（无需 Slash 指令）

当出现 DM 失败/一致性异常时，系统会在“投票面板频道”或“报告频道”发送一次性错误报告消息（Embed）。

处理方式：

1) 按照消息中的“处理建议”完成人工操作（例如：手动通知成员、手动移除残留身份组等）
2) 点击该消息下方按钮：**✅ 标记为已处理**

验证点：

- 报告消息按钮被禁用
- sr_system_alerts 对应记录写入 resolved_at
- 若该告警关联 grant 且该 grant 已无其它未解决告警，`manual_attention_required` 会被自动清除

---

## 9. 一致性巡检（面板丢失 / 结束 grant 角色残留）

### 9.1 面板消息丢失

1) 手动删除用户面板消息
2) 执行：`/自助身份组申请-测试 执行一致性巡检`

验证点：

- 写入告警：`panel_message_missing`
- 该面板记录被标记 inactive（避免后续继续尝试刷新不存在的面板）

### 9.2 ended grant 但角色仍残留

制造方式（推荐）：

1) 让用户“退出”或“强制清退”结束 grant
2) 管理员手动把岗位角色重新加回该用户（制造残留）
3) 执行：`/自助身份组申请-测试 执行一致性巡检`

验证点：

- 写入告警：`ended_grant_roles_still_present`
- 若配置了 reportChannel，会在报告频道提示管理员手动移除

---

## 10. 重置与复测建议

为了保证测试可重复，建议每轮大测试后做一次清理：

- 删除测试面板消息（或直接重召新的面板）
- 清理测试账号身上的岗位身份组/配套身份组
- 如需彻底重置 SelfRole 数据：
  - 停止 bot
  - 删除 `data/selfRole.sqlite`
  - 重新启动 bot（会自动重建表结构）

如果你频繁切换项目/命令集合，并且出现“旧命令残留”，建议在测试环境短期开启一次：

- `CLEAR_GLOBAL_COMMANDS=true`

## 11. 常见问题排查

1) 命令没出现/不是最新：
- 确认 `.env` 的 `GUILD_IDS` 包含当前测试服
- 建议开启 `STRICT_COMMAND_SYNC=true` 观察启动期同步日志
- Discord 客户端有缓存时，等待片刻或重启客户端

2) 旧命令仍存在：
- 如果是该应用的全局命令遗留，开启 `CLEAR_GLOBAL_COMMANDS=true` 重启
- 全局命令删除可能延迟生效（约 1 小时）

3) 授予/移除身份组失败：
- 检查机器人是否有 Manage Roles 权限
- 检查机器人最高身份组是否高于目标岗位身份组/配套身份组

4) DM 相关测试始终失败：
- 确认测试用户允许接收来自服务器成员的私信
- 失败也应形成告警并给出管理员指引（这本身就是要测试的能力）

---

## 12. 退出测试模式

1) 关闭测试命令注册：

```bash
SELF_ROLE_ENABLE_TEST_COMMANDS=false
```

2) 重启 bot

验证点：

- `/自助身份组申请-测试` 从命令列表中消失
