# 自助身份组综合管理模块（selfRole）

> 本文档已按“综合管理系统重构（Stepwise v3）”更新。

## 1. 模块定位

本模块用于在服务器内提供“身份组岗位”式的综合管理能力：

- 用户通过 **用户面板** 自助申请身份组
- 管理员可为每个身份组配置：
  - 前置条件（前置身份组、频道活跃度、活跃天数等）
  - 人数上限（容量）与“名额预留”
  - 是否进入人工/社区投票审核
  - 通过后自动授予“配套身份组（bundle roles）”
  - 周期询问留任/退出 + 强制清退（仅管理 bot 发放的成员）
- 系统具备：
  - **申请生命周期**（pending 预留名额、7天过期释放、用户撤回）
  - **grant 生命周期**（记录授予时间、后续周期询问/强制清退）
  - **告警系统**（DM 失败、异常一致性）
  - **一致性巡检**（结束 grant 但角色仍残留、面板丢失等）

> 重要边界：本模块**只管理通过本模块发放的身份组（grant 记录）**。
> 
> 服务器中“手动给予/其它系统给予”的同名身份组成员，不会被纳入周期清退与一致性修复范围。

## 2. 存储架构（当前）

### 2.1 SQLite（主存储）

SelfRole 主库位于：

- `data/selfRole.sqlite`

核心表（简述）：

- 配置与活跃度（legacy + 兼容）：
  - `role_settings`（每个 guild 一行，roles 为 JSON 字符串）
  - `user_activity` / `daily_user_activity`
  - `role_applications`（legacy 投票流程正在使用；仅存储进行中的投票）
  - `role_cooldowns`

- v2 运行态表（重构主线）：
  - `sr_applications_v2`：申请生命周期（pending/withdrawn/expired/approved/rejected 等；支持 reserved_until）
  - `sr_grants`：grant 生命周期（谁在何时被 bot 授予、下一次询问/强制清退时间、是否需要管理员介入等）
  - `sr_grant_roles`：grant 对应的主身份组 + 配套身份组
  - `sr_renewal_sessions`：留任询问 session（DM 按钮交互）
  - `sr_panels`：面板注册（用于停用旧面板、刷新统计、巡检）
  - `sr_system_alerts`：系统告警（DM 失败、一致性异常等）

### 2.2 JSON

本项目其它模块仍存在 JSON 存储；SelfRole 当前核心数据已以 SQLite 为主。

## 3. 核心流程说明

### 3.1 用户申请流程（含名额预留）

1. 用户点击 **用户面板** 按钮（新面板按钮 customId=`sr2_apply_button`；旧按钮 `self_role_apply_button` 仍兼容但会提示尽快重召面板）
2. 系统展示可申请身份组列表（已满员的身份组不会出现在列表）
3. 对每个选择的身份组进行条件校验：
   - 前置身份组
   - 频道活跃度
   - 活跃天数
   - 冷却期
   - 容量校验：**现任人数 + v2 pending 预留名额**
4. 若需要审核：
   - 创建 v2 申请记录（`sr_applications_v2`，slot_reserved=1，reserved_until=now+7d）
   - 在审核频道创建投票面板（legacy `role_applications` 仍负责投票数据）
5. 若无需审核（直授）：
   - 一次性授予【主身份组 + 配套身份组】
   - 写入 grant：`sr_grants` + `sr_grant_roles`

### 3.2 pending 过期释放（默认 7 天）

- pending 默认 7 天过期
- 过期后：
  - v2 application 标记为 expired 并释放 slot_reserved
  - legacy 投票记录被删除并禁用面板按钮
  - 用户面板统计会自动刷新

### 3.3 用户撤回申请

用户命令：

- `/自助身份组申请-撤回申请`

效果：

- 将 v2 pending application 标记为 withdrawn 并释放名额
- 禁用审核面板按钮（避免继续投票）
- 刷新用户面板统计

### 3.4 审核通过/拒绝

- 审核通过：
  - 授予【主身份组 + 配套身份组】
  - 写入 grant 记录
  - v2 application 终结并释放名额

- 审核拒绝：
  - 写入冷却期（若配置）
  - v2 application 终结并释放名额

## 4. 周期询问与强制清退（仅 grant 追踪对象）

### 4.1 配置入口

使用：

- `/自助身份组申请-配置身份组 周期清退配置`

可配置字段（存储于 roleConfig.lifecycle）：

- `enabled`：是否启用周期管理
- `inquiryDays`：每隔 N 天询问留任/退出（0=不询问）
- `forceRemoveDays`：授予后 N 天强制清退（0=不强制）
- `onlyWhenFull`：是否仅在“满员（含 pending 预留名额）”时执行询问/清退
- `reportChannelId`：报告频道（发送询问前/询问后/清退前/清退后报告 + DM 失败提示）

### 4.2 onlyWhenFull 语义

- 计时继续累积（next_inquiry_at/force_remove_at 以授予时间为基准）
- **未满员时不执行**询问/清退动作
- 当达到满员时：若已逾期，可能立即触发，并附带“延后触发”说明文案

---

## 5. 异常报告（DM 失败 / 一致性异常）

- 异常追踪（用于去重/排查）：`sr_system_alerts`
- grant 需要管理员介入（内部标记）：`sr_grants.manual_attention_required = 1`

处理方式：

- 系统在报告频道/相关频道发送**一次性显眼错误报告**（Embed）
- 报告消息附带按钮：**✅ 标记为已处理**（无需使用 Slash 指令）
- 可选：通过环境变量 `SELF_ROLE_ALERT_MENTION_ROLE_ID=<roleId>` 指定要 @ 的管理员身份组（不配置则不 ping）

---

## 6. 一致性巡检（MVP）

后台定时任务会：

- 巡检 `sr_panels` 的 active 面板是否仍存在；丢失则自动标记为 inactive 并写入告警
- 巡检近 30 天内结束的 grant：若成员仍持有 grant 关联角色，则写入高优先级告警，并在 reportChannel（若有）提醒管理员手动移除

## 7. 管理与用户命令（概要）

### 7.1 面板

- `/自助身份组申请-创建自助身份组面板`
- `/自助身份组申请-创建管理面板`

用户面板支持两种展示方式：

- **简洁模式（默认）**：静态说明 + 在同一 embed 中维护 `【岗位状态】` 区块
- **详细模式**：静态说明 + 额外附带两个动态区块（独立 embeds）：
  - `【岗位状态】`
  - `【岗位详情】`（展示条件/名额/是否审核/活跃度等）

创建时可在 `/自助身份组申请-创建自助身份组面板` 中使用参数：`详细模式:true`

### 7.2 配置（推荐使用）

- ✅ 推荐：`/自助身份组申请-配置向导`
  - 用多步交互一次性完成：审核/直授、人数上限、前置身份组、活跃度、配套身份组、周期清退、申请理由等
  - 审核频道/报告频道支持设置为“子区（线程）”；如果下拉选不到，可使用按钮通过粘贴链接/ID 设置。

- `/自助身份组申请-配置身份组`
  - 基础配置（含：描述、活跃度、**人数上限**）
  - 审核配置
  - 配套身份组配置
  - 周期清退配置
  - 申请理由配置
  - 移除配置
  - 展示已配置身份组

> 小提示：`审核配置` 支持 `审核频道` 或 `审核频道链接`（二选一）；`周期清退配置` 支持 `报告频道` 或 `报告频道链接`。

> 说明：旧的交互式管理面板（adminPanelService 的 modal）目前未覆盖“人数上限/配套身份组/周期清退”等新字段；建议使用上面的统一配置命令进行配置。

### 7.3 用户操作

- `/自助身份组申请-撤回申请`

### 7.4 运维

- 无需告警 Slash 指令：直接在“错误报告消息”上点击 **✅ 标记为已处理** 即可。
- `/自助身份组申请-运维`
  - 刷新面板 / 检查过期 / 强制过期 / 执行生命周期 / 查看 grant / 执行一致性巡检 等

## 8. 兼容与迁移注意事项

- 旧用户面板按钮 `self_role_apply_button` 仍可触发申请，但会提示尽快重召新面板。
- 启动时会将 legacy pending 投票（`role_applications.status='pending'`）自动迁移到 v2（用于“名额预留/统计口径接管”）。
- **名额统计口径（最新规则）**：
  - 现任人数 = 本地 grant 记忆（`sr_grants/sr_grant_roles` 中 active 的记录）
  - 待审核人数 = v2 pending 预留名额（`sr_applications_v2.slot_reserved=1` 且 reserved_until 未过期）
  - 服务器内手动授予/其它系统授予的同名身份组成员：
    - **不会占用名额**
    - **不会被周期清退**
- manual takeover grant：当前决定**不实现**（主线已可通过运维命令与告警机制完成日常运营）。
- richer panel display：已支持“详细模式”（`/自助身份组申请-创建自助身份组面板 详细模式:true`），后续如需更复杂排版再追加模板能力。

## 9. 灰度上线 / 迁移推荐步骤

1. 拉取新版本代码并重启 bot（启动时会自动初始化/演进 `data/selfRole.sqlite` 表结构）。
2. （可选）使用 `/自助身份组申请-配置身份组 展示已配置身份组` 检查现有配置是否正常。
3. 在公开报名频道重新执行：`/自助身份组申请-创建自助身份组面板`
   - 新面板按钮为 `sr2_apply_button`
   - 系统会尝试停用 DB 中记录的旧面板（历史未记录面板无法自动停用）
4. 为需要“满员才执行 onlyWhenFull”的岗位配置人数上限：
   - `/自助身份组申请-配置身份组 基础配置 人数上限:<N>`
5. （如需）配置配套身份组与周期清退：
   - `/自助身份组申请-配置身份组 配套身份组配置`
   - `/自助身份组申请-配置身份组 周期清退配置 报告频道:<#channel>`（或使用 `报告频道链接` 以设置到子区/线程）
6. 上线后关注：
   - 报告频道消息（DM 失败/一致性异常会发送一次性错误报告，带“✅ 标记为已处理”按钮）
